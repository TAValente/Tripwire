# Anti-Patterns

Tripwire should avoid:

- naming debates
- formatting complaints
- brace-style discussions
- generic best-practice lectures
- low-value nitpicking
- maximizing finding count at the expense of accuracy
- interrupting the user when the interruption does not improve project speed
- acting certain when it is only trying to calibrate judgment

Project anti-patterns to flag:

- architectural drift
- hidden complexity
- premature abstractions
- requirement mismatches
- unit economics regressions
- inconsistent decisions
- accidental scope expansion
- making optional infrastructure mandatory
- adding product surface before reviewer quality is validated
- treating AI output as authoritative when local doctrine or guardrails disagree
- storing data without a clear validation purpose
- applying Tripwire's own doctrine to a different project under review
